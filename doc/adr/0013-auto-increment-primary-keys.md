# ADR-0013: Auto-increment primary keys (sequence portability scope)

## Status

Accepted (2026-05-08)

## Context

Auto-incrementing integer IDs are the most common primary-key pattern. Both target backends support them, but with very different surface area:

- **PostgreSQL** offers standalone `SEQUENCE` objects (`CREATE SEQUENCE`, `nextval`, custom start, increment, min/max, cycle, cache, ownership, sharing across tables), legacy `SERIAL`/`BIGSERIAL` shortcuts, and modern identity columns (`GENERATED BY DEFAULT AS IDENTITY` / `GENERATED ALWAYS AS IDENTITY`).
- **SQLite** offers only `INTEGER PRIMARY KEY AUTOINCREMENT` — a per-table hidden sequence stored in `sqlite_sequence`. There is no standalone sequence object, no custom start/increment, no sharing across tables.

The portable intersection is "this integer primary key auto-increments." Anything beyond that is Postgres-only.

A v1 that exposes the full Postgres sequence surface drops portability. A v1 that exposes nothing leaves out a feature every user expects on day one. The right cut for v1 is the intersection — and to leave the DSL space for a future Postgres-only standalone sequence object without committing to it now.

## Decision

v1 supports auto-incrementing primary keys via a single field-level annotation:

```crystal
class User < Prostore::Model
  field 1, :id, Int64, primary: true, auto_increment: true
end
```

### Rules

1. **`auto_increment: true` is the only sequence-related knob in v1.** Renders as `GENERATED BY DEFAULT AS IDENTITY` on PostgreSQL and `INTEGER PRIMARY KEY AUTOINCREMENT` on SQLite.

2. **`GENERATED BY DEFAULT AS IDENTITY`, not `ALWAYS`.** The `BY DEFAULT` flavor permits manual values to be inserted (useful for backfills that copy explicit IDs from legacy data). `ALWAYS` rejects manual insertion entirely and forces `OVERRIDING SYSTEM VALUE` clauses, which we don't want in regular code paths.

3. **Allowed only on integer primary keys.** Specifically `Int32` or `Int64`. Compile error otherwise. The intersection of what both backends support — `Int8`/`Int16` overflow too easily and aren't commonly available as identity types.

4. **Adding or removing `auto_increment` requires a new tag.** Per ADR-0003, this is a column-shape change. Strict.

5. **Sequence resync after explicit-ID backfills.** When a backfill writes explicit values to an `auto_increment` column (typical for v1→v2 ID migrations where IDs are copied from a legacy column), the planner appends a `reset_sequence` step. On Postgres this issues `SELECT setval(pg_get_serial_sequence('<table>', '<col>'), (SELECT max(<col>) FROM <table>))`. On SQLite, `sqlite_sequence` is updated by the engine when the explicit values exceed the previous max, so no explicit step is required, but the planner still emits a no-op marker for symmetry in the migration log.

6. **Sequence current values are not tracked in `prostore_schema`.** Current value is runtime data, not schema. Drift detection (ADR-0010) ignores it. If a DBA manually advances or rewinds a sequence, the library does not interfere.

7. **No standalone sequence object in v1.** Custom Postgres sequences (different start, increment, shared across tables, monotonic-only, not bound to a specific column) are out of scope. The DSL keyword `sequence` is reserved for a future ADR; using it raises a compile error pointing at this ADR.

## Consequences

- The portable v1 surface is "auto-incrementing integer primary key" — what most apps need and nothing more.
- Backfills that copy explicit IDs from legacy columns work without next-insert collisions, because the planner emits `reset_sequence` after the backfill step.
- The choice of `BY DEFAULT` over `ALWAYS` on Postgres allows backfills and tests to write explicit IDs without ceremony.
- Operators wanting custom Postgres sequences (sharing, custom increment, distributed-ID generation) cannot get them through prostore in v1. They can manage them outside prostore — but doing so creates unmanaged state per ADR-0010.
- The `sequence` DSL keyword is reserved, so a future v2 introducing standalone sequence objects does not need to break the DSL surface.
- `auto_increment` on a non-integer or non-PK column is caught at compile time, before any DDL runs.

## Alternatives Considered

- **`GENERATED ALWAYS AS IDENTITY` on Postgres.** Rejected: blocks manual insertion of explicit IDs without an `OVERRIDING SYSTEM VALUE` clause. Inconvenient for backfills, tests, and any ID migration. The "always" flavor's safety benefit (preventing accidental manual insertion) is small compared to the friction it adds to legitimate paths.
- **`BIGSERIAL` on Postgres.** Rejected: legacy syntax. Identity columns are the modern recommendation and produce equivalent behavior with cleaner introspection. Existing `BIGSERIAL` databases work identically with the identity-column form because the underlying mechanism (a sequence) is the same.
- **First-class standalone sequence object in v1.** Rejected: scope creep. Postgres-only, niche compared to the auto-increment use case, and committing to a DSL surface for it before working through actual use cases would lock us into something we'd want to revise. Reserving the keyword preserves the option without paying for it now.
- **Skip `reset_sequence` after backfills.** Rejected: silent next-insert collisions are a serious footgun. The cost of one extra step per backfill that touches an `auto_increment` column is trivial compared to the debugging cost of a unique-violation hours after the migration ostensibly succeeded.
- **Allow `auto_increment` on non-PK columns.** Rejected: SQLite can't express it (AUTOINCREMENT is bound to `INTEGER PRIMARY KEY`). Permitting it would silently produce different semantics across backends.
