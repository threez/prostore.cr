# ADR-0010: Drift detection and correction

## Status

Accepted (2026-05-08)

## Context

ADR-0001 makes the model the source of truth for the desired schema. In practice, the live database can diverge from that truth: a DBA hand-runs DDL, another tool partially applies a migration, an emergency hotfix bypasses the library, or two prostore versions briefly disagree during a deploy. The library has to decide what to do when the live DB doesn't look like the library's last record of it.

Three policies are conceivable: refuse to operate against any drift, silently overwrite to match the model, or split by repairability — fix what can be fixed without violating the design constraints, error on what cannot. Refusal is operationally hostile (every drift becomes an outage). Overwriting risks silent data loss (re-creating a dropped column writes an empty column where user data lived). The split policy aligns with the rest of the design: ADR-0003 already forbids in-place type changes and that constraint applies equally to drift correction.

Detecting drift in a way that respects ADR-0002 (stable tags) requires the library to remember the mapping between tags and live database objects. Column names alone aren't enough — a renamed column that hasn't been recorded looks identical to a drop+add. The library needs a third bookkeeping table alongside the migration tables from ADR-0009.

## Decision

The library detects drift at the start of every migration run and corrects what is fixable as part of the same migration plan. Unfixable drift aborts the run before any DDL is applied.

### Prerequisite: schema bookkeeping table

```sql
CREATE TABLE prostore_schema (
  table_name    TEXT NOT NULL,
  kind          TEXT NOT NULL,   -- 'column' | 'index'
  tag           INT  NOT NULL,
  current_name  TEXT NOT NULL,   -- name in the live DB, updated after each migration step
  definition    TEXT NOT NULL,   -- JSON: type, nullability, predicate, etc.
  PRIMARY KEY (table_name, kind, tag)
);
```

This table is updated atomically at the end of each successful migration step. It is the library's record of "what we believe the live DB looks like, by tag."

### Drift detection

At the start of a migration run, the library reads `prostore_schema`, queries live DB metadata (`information_schema` on Postgres, `sqlite_master` + `pragma_table_info` on SQLite), and diffs them. The result is partitioned into:

- **Fixable drift**: discrepancies the library can reconcile without violating ADR-0003 or risking user data.
- **Unfixable drift**: discrepancies that would require a forbidden operation or imply data loss.
- **Unmanaged state**: live DB objects that have no corresponding entry in `prostore_schema`.

### Resolution by case

| Case | Resolution |
| ---- | ---------- |
| Managed column renamed externally | Auto-fix: rename back to expected name |
| Managed column dropped externally | **Error** — data is gone; operator must restore or reserve the tag |
| Managed column type / nullability altered | **Error** (ADR-0003) |
| Managed column has extra constraint not in model | **Error** (e.g., a `NOT NULL` we didn't apply) |
| Managed index dropped or definition altered | Auto-fix: drop (if present) and re-create from model |
| Managed index renamed externally | Auto-fix: rename back |
| Unmanaged column / index / table | Leave alone; report at startup |
| Unmanaged object name collides with a desired tag | **Error** — operator must resolve |
| `prostore_schema` missing | Treat as first run; re-discover and populate from live DB |
| `prostore_schema` inconsistent in a way drift detection cannot explain | **Error** — operator intervention required |

The asymmetry between dropped columns (error) and dropped indexes (auto-fix) is by data: indexes carry no data, columns carry user data. Silently re-adding a dropped managed column would write an empty column where data lived. Failing loudly is the right default.

### Integration with the migration state machine

Fixable drift becomes prepended steps in the migration plan generated by ADR-0009's planner — `rename_column_back`, `recreate_index`, etc. — recorded in `prostore_migration_step` alongside model-driven steps. Resumability, leasing, and abort semantics carry over unchanged. An operator inspecting `prostore migrate status` sees drift correction as ordinary steps with descriptive `kind` values.

A migration runs even when the model has not changed if drift is present. The plan in that case contains only drift-correction steps. A `prostore drift check` CLI dry-runs the detection without applying anything.

## Consequences

- The model-as-truth philosophy is enforceable, not aspirational. The library will not operate against state it cannot reconcile, but it will repair the cases it safely can.
- Adding `prostore_schema` brings the bookkeeping count to three tables (`prostore_migration`, `prostore_migration_step`, `prostore_schema`). The reserved name prefix continues to apply.
- Operators get a clear error message when drift is unrepairable, including the specific tag and discrepancy, rather than a confusing migration failure deep in DDL.
- Unmanaged state coexists. Hand-added columns or indexes do not break operation; they are reported at startup and otherwise ignored. A future strict-mode flag could refuse to operate with any unmanaged state present, but is out of scope for v1.
- Drift correction increases the surface area of the planner: every drift case has to be expressible as one or more atomic steps with the same idempotency contract as model-driven steps. This is achievable because the fixable cases (renames, index recreation) are mechanical operations the engine already supports.
- The `current_name` column in `prostore_schema` makes it possible to track names across renames without loss. A rename in the model produces a step that updates both the live DB and `current_name` atomically; drift detection compares against `current_name`, not the model name.

## Alternatives Considered

- **Refuse to operate against any drift.** Rejected: every drift becomes an outage. Common cases like a renamed managed column or a dropped managed index are mechanically fixable and should not require operator intervention.
- **Silently overwrite live DB to match model.** Rejected: re-adding a dropped managed column produces an empty column where user data lived. Silent data loss is unacceptable.
- **Per-table drift policy** (operator can declare some tables strict, others tolerant). Rejected for v1: more configuration surface than warranted before we have evidence the global policy is wrong.
- **Detect drift only on demand via a CLI**, not automatically before each migration. Rejected: skipping detection means a migration run can fail mid-DDL with confusing errors. Detect-before-plan is cheap and catches problems early.
- **Auto-repair `prostore_schema` corruption from the live DB.** Rejected: if the bookkeeping is corrupt, the library has no reliable way to know which tag a live column belongs to. Repairing by inference would silently legitimize whatever state the DB is in. Operator intervention is the only safe path.
